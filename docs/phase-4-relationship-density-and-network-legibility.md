# Phase 4 - Relationship Density and Network Legibility

## Why Phase 4 exists

Phase 3 proved that one doctrine topic can hold multiple views, assessments, and relationship objects without collapsing identity and judgment together.

Phase 4 exists to make the network more visibly useful by increasing connection density and making the network easier to read for maintainers, scholars, and future AI systems.

## What belongs in Phase 4

- additional relationship objects for all major pilot views
- denser grounding links from views to passages
- denser links from views to assessments
- relationship manifests that summarize a doctrine network at a glance
- human-readable guidance for how to interpret a doctrine network

## Why density matters

A sparse graph proves structure.
A denser graph proves advantage.

Once multiple views have explicit grounding, assessment, contrast, and neighboring relations, the repository starts to function less like a note archive and more like a governed theological network.

## Why legibility matters

Relationship density alone is not enough.

If a user cannot tell what a topic is, what a view is, what an assessment is, and what a relationship object is doing, then the graph becomes impressive but opaque.

Phase 4 therefore pairs network growth with reading guidance.

## What users should now be able to see

Users should be able to see:

- that one doctrine topic can contain multiple neighboring and conflicting views
- that assessments belong to views under a baseline rather than to the doctrine topic itself
- that scriptural grounding can be attached to specific views as explicit relationship objects
- that some views are closer neighbors while others are sharper contrasts
- that the repository can scale through repeated network patterns rather than ad hoc prose accumulation

## What Phase 4 does not yet do

Phase 4 does not yet provide:

- weighted coherence scoring
- automated contradiction or dependency analysis
- formal validation contracts for every relationship type
- scholar workflow automation or review queues

## Definition of done for Phase 4

Phase 4 is done when:

- all major Trinity pilot views have explicit assessment links
- at least the main orthodox and neighboring views have explicit grounding links
- the doctrine network can be read through a compact manifest and a human-readable guide
- the repository shows obvious structural advantage over flat note-based doctrine files
