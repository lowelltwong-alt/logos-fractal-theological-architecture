# inst.alpha

## Purpose

This file is the first coded institutional overlay draft built from the Logos upstream architecture.

It is intentionally written as a reusable coded instantiation rather than a public-facing case memo tied to a named institution.

It should be read together with:
- `docs/governance/tradition-composition-profile.md`
- `docs/governance/institution-overlay.md`
- `docs/governance/hitl-decision-rules.md`
- `docs/governance/build-priority-policy.md`
- `docs/airca/README.md` in the companion AIRCA repository

## Overlay identity

### Coded name
`inst.alpha`

### Institution type
A distributed mission-driven service network.

### Institutional pattern
This institution operates through a mixed environment that includes:
- board governance
- executive leadership
- internal staff operations
- distributed local or affiliate execution
- donor and supporter visibility
- beneficiary-facing mission delivery

## Inherited upstream profile

`inst.alpha` should inherit from a Christian tradition composition profile that strongly emphasizes:
- human dignity
- stewardship
- participation
- institutional trust
- moral accountability
- justice and service
- non-deformation under technological convenience

At this stage, the exact formal tradition profile may remain provisional.
The important point is that `inst.alpha` is downstream from a reusable theological source architecture rather than from a local operating memo alone.

## Mission shape

The mission pattern of `inst.alpha` is:
- direct service under a moral frame
- institutionally visible public trust obligations
- mixed stakeholder translation needs
- recurring pressure to improve scale, consistency, and throughput without reducing persons to process objects

In plain terms, `inst.alpha` must preserve dignity while coordinating real-world delivery.

## Governance environment

The main governance pressures on `inst.alpha` include:
- board-level concern for mission integrity and risk
- executive pressure for scale, consistency, and implementation
- need for staff trust and adoption
- sensitivity to public reputation
- need to preserve beneficiary dignity and participation
- recurring translation between theological, strategic, and operational language

## Stakeholder map

The main stakeholder groups are:
- board
- executive leadership
- strategy / chief-of-staff type operators
- staff
- distributed local operators or affiliates
- beneficiaries or participant households
- donors and supporters
- external partners

## Recurring decision classes

The recurring decisions most relevant to `inst.alpha` include:

### 1. Mission-coherence decisions
Examples:
- whether a new capability fits the mission
- whether efficiency logic is overtaking dignity logic
- whether scale proposals preserve institutional trust

### 2. Workforce and capability decisions
Examples:
- how AI-assisted workflows affect staff roles
- what should be automated, augmented, or kept relational
- what capability-building is required before deployment

### 3. Service-design decisions
Examples:
- whether a workflow increases access while preserving agency
- whether beneficiaries are being treated as participants or throughput objects
- what level of human contact must remain visible

### 4. Orchestration and coordination decisions
Examples:
- what should be standardized centrally
- what should remain local or affiliate-specific
- where coordination yields leverage without erasing local judgment

### 5. Governance translation decisions
Examples:
- how to translate theology into board-safe language
- how to keep theological inheritance visible without making every downstream artifact sound identical

## Mission-constrained translation layer

`inst.alpha` should translate upstream theology into language such as:
- mission integrity
- dignity and participation
- prudent stewardship
- public trust
- accountable coordination
- human-centered implementation

This translation should not sever the theological source from the institutional language.
It should render it operationally legible.

## HITL implications

For `inst.alpha`, the following should normally require at least `review_required` or stronger:
- any workflow that materially changes how beneficiaries experience agency
- any proposal that reduces human contact in mission delivery
- any workforce redesign tied to AI-assisted systems
- any institution-wide standardization proposal that reduces local discretion

The following should normally require `escalation_required` or stronger:
- changes to governance-facing AI rules
- changes to trust or review classifications
- institution-wide adoption of AI-assisted decision support in sensitive service environments
- changes that could reshape public trust or mission identity

The following should normally require `authority_commit_required`:
- adoption of formal institution-wide AI operating principles
- adoption of formal use-case prohibitions or permissions
- decisions that materially alter how dignity, participation, or mission delivery are interpreted at scale

## Prohibited drifts

`inst.alpha` should guard against:
- efficiency-first redesign that weakens dignity
- shallow automation that reduces participation
- role compression without capability growth
- centralization that treats local operators as passive implementers only
- governance language that hides real theological commitments rather than translating them faithfully

## Suggested next derived artifacts

The strongest next downstream artifacts for `inst.alpha` would be:

## Summary principle

`inst.alpha` is a coded overlay for a distributed mission-driven service institution that needs to scale coordination and decision quality without sacrificing dignity, participation, trust, or visible human accountability.
