# Retrieval and Agent-Use Principles

## Purpose

This file defines how the repository should be shaped for future retrieval systems, graph traversal, and AI-agent use.

This file should be read together with:
- `operational-ontology-model.md`
- `research-synthesis-ai-era-ontology-takeaways.md`
- `asserted-inferred-and-boundary-layers.md`
- `controlled-schemes-and-vocabularies-policy.md`

## Core principle

A future-ready ontology is not designed only for human reading or formal reasoning.

It is also designed for:
- semantic retrieval
- graph traversal
- context packaging
- trust-aware filtering
- structured AI grounding

## What this means for the repository

The repository should increasingly optimize for:
- stable identifiers
- rich labels and alternative labels
- strong short definitions
- typed relationships
- bridge links between text, concepts, doctrines, thinkers, and witnesses
- context-aware notes rather than only long undifferentiated prose

## Human-readable and machine-legible rule

The repository should remain human-readable first, but machine-ingestible by design.

That means a strong node should usually offer both:
- readable explanatory prose
- a stable machine-usable structure

## Retrieval rule

A retrieval system should be able to use the repository to answer questions without guessing too much from raw prose alone.

That becomes easier when the repository includes:
- stable node IDs
- direct local links
- scheme memberships
- clear boundary or trust indicators
- provenance and review state

## Agent-use rule

A future agent should be able to determine:
- what kind of object something is
- whether it is asserted or inferred
- what trust zone applies
- what scheme or vocabulary governs it
- whether it is safe to reuse or only safe to inspect

A future agent should not have to infer all of that from vague wording alone.

## Bridge-object rule

The repository should continue building bridge objects that help connect levels.

Examples include:
- relationship objects
- concordance objects
- reconstruction objects
- translation comparison objects
- lexical evidence objects

These are high-value because they connect narrative material to machine traversal.

## Retrieval summary rule

Over time, the repository may also benefit from compact retrieval-oriented summaries or graph-neighborhood summaries.

Those should not replace the underlying nodes.
They should be derived, auditable, and tied back to stable objects.

## Trust-filter rule

Future retrieval should be able to filter by trust and boundary context.

Examples:
- retrieve only core trusted nodes
- include reviewed specialized lexical or witness material
- exclude quarantine or boundary-restricted material by default
- include one tradition-boundary layer intentionally

## Fractal rule

Retrieval-aware design should recur across scales.

That means the same general logic should apply whether the object is:
- a small lexical node
- a doctrine hub
- a witness object
- a graph domain
- a larger export package

## Summary principle

The future ontology should help humans read well and help machines traverse responsibly.

That is one of the highest-leverage design goals in the AI era.
