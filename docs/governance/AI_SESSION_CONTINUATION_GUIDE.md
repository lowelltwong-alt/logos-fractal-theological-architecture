# AI Session Continuation Guide

## Purpose

This document is a handoff and continuation guide for future AI chat sessions, agents, and collaborators working on the Logos Fractal Theological Architecture repository.

It explains:
- what this repository is trying to become
- what design decisions have already been made
- how the recursive / fractal / DNA-like architecture works
- where the self-reinforcing structure should live in the repository
- how to safely improve and extend the system
- what should be preserved while the repository becomes more automated and more machine-legible

## Repository Position

This repository is a governed theological source architecture.

It is meant to become:
- recursive
- fractal
- ontology-ready
- retrieval-aware
- trust-aware
- review-aware
- AI-assistable
- extensible across Christian shared-core, tradition-specific, and institution-specific use

It is not merely a document library, note vault, or disconnected graph experiment.

## What Must Be Preserved

Future sessions should preserve these high-level commitments unless intentionally changed through governance:
- stable identity
- stable anchors
- separation of identity and structural address
- explicit trust and boundary logic
- human-authored local truth with machine-derived larger traversal
- doctrinal seriousness
- noncanonical boundary discipline
- recursive self-similarity across repository branches
- promotion by governed thresholds rather than uncontrolled duplication

## Core Design Decisions Already Made

### 1. Stable identity matters
Existing semantic IDs and anchors should not be casually renamed.

### 2. Address must be separate from identity
The repository should maintain stable semantic IDs while adding a structural address field.

### 3. AI may accelerate buildout but not silently self-authorize theology
AI may propose, normalize, validate, and queue. Humans remain responsible for doctrinal approval, trust promotion, and disputed theological judgment.

### 4. Claims should become first-order governed objects
The repository should not rely only on prose and naked graph edges.

### 5. Shared core and overlays must remain distinguishable
The repository should support shared Christian core material while allowing governed overlays for traditions, institutions, and applications.

### 6. Validation is part of the architecture
Validation and CI should become part of the repository DNA rather than optional tooling garnish.

## Build Order

Future sessions should improve this repository in the following order:

1. reconcile and stabilize governance
2. define the structural address system
3. define the claim layer
4. normalize root metadata shells
5. define validation and CI policy
6. define feedback and drift-detection loops
7. build one full reference implementation slice
8. add overlay logic
9. expand content gradually
10. automate more aggressively only after the earlier layers are stable

## What Future AI Sessions May Do

AI may safely help with:
- metadata normalization
- address suggestions
- schema drafting
- duplicate detection
- reference checks
- migration record drafting
- proposal drafting
- review queue generation
- retrieval neighborhood packaging

## What Future AI Sessions Must Not Do Silently

AI must not silently:
- redefine doctrine
- promote speculative claims into core status
- erase boundary distinctions
- create new canonical authorities
- flatten asserted, inferred, and hypothetical claims together
- change stable identities without migration records
- treat draft material as reviewed truth

## Working Rule

The correct sequence is:
- reconcile first
- validate second
- expand third
- automate fourth

If uncertainty exists, prefer:
- fewer canonical files
- clearer governance
- stronger review posture
- explicit lineage
- safer migration discipline

over premature expansion.
