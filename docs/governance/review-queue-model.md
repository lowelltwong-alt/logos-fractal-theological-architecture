# Review Queue Model

## Purpose

This file defines how proposals should be routed for human review.

## Core principle

Not every proposal carries the same sensitivity.
The review queue should reflect that.

## Queue families

### 1. Metadata and normalization queue
For low-risk structural work such as field completion, tag normalization, and address suggestion review.

### 2. Branch-content queue
For ordinary branch buildout such as new doctrine, concept, scripture, lexical, or manuscript nodes.

### 3. Claim-review queue
For claim objects, especially where evidence or epistemic posture needs confirmation.

### 4. Boundary and trust queue
For noncanonical classification, trust-zone promotion, overlay-scope disputes, and related high-sensitivity cases.

### 5. Synthesis and governance queue
For major synthesis claims, new relationship verbs, and governance changes.

## Routing rule

A proposal should be routed according to:
- branch
- sensitivity
- trust implications
- doctrinal impact
- whether the change affects shared core or overlays

## Summary principle

Review queues should make human oversight more scalable by concentrating attention where risk and theological significance are highest.
