# Phase 1 — Constitutional Core

Phase 1 locks the non-negotiable operating grammar of the repository.

It does four things:

1. freezes the constitutional vocabulary
2. freezes trust-zone semantics
3. defines promotion authority
4. defines the minimum repeatable module contract for future buildout

## Phase 1 exit criteria

Phase 1 is complete only when all of the following are true:

- doctrine topic, doctrine view, and doctrine assessment are separate first-class objects
- asserted, inferred, proposed, and learning-sidecar material are structurally separated
- every promoted object can declare status, trust zone, provenance, and review state
- human approval gates are defined for canonical promotion and schema change
- future modules can inherit a repeatable folder and validation pattern

## Scope freeze

The following concepts are frozen for v0.1 and may not be casually redefined:

- DoctrineTopic
- DoctrineView
- DoctrineAssessment
- TraditionProfile
- PassageInterpretation
- Hermeneutic
- Assertion
- CoherenceEdge
- TrustZone
- ReviewState
- LifecycleStatus

## Phase 1 does not do everything

Phase 1 does not populate the full doctrine corpus.
It does not finalize the full coherence engine.
It does not settle every manuscript or source-critical branch.
It establishes the constitutional and governance spine that later layers must obey.

## Constitutional rules

- identity is not label
- identity is not path
- lower-trust material may not silently overwrite higher-trust material
- the coherence engine is a sidecar, not canon
- human approval is required for canonical promotion, boundary adjudication, orthodoxy judgments, and active schema change
- nothing important is deleted from history; deprecated material remains resolvable
