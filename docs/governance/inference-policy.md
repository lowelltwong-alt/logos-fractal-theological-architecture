# Inference Policy

## Purpose

This file defines the boundary between what humans should author directly and what machines may infer from governed local assertions.

A fractal ontology becomes much stronger when humans are not forced to hand-author every global relationship and machines are not allowed to invent meaning beyond governed structure.

This file should be read together with:
- `fractal-rules.md`
- `competency-questions.md`
- `relationship-registry.md`
- `schema-compatibility-policy.md`

## Core principle

Humans should author local, immediate structure.
Machines may derive larger recursive structure from governed local assertions.

## Human-authored layer

Humans should normally author:
- stable node identity
- node type and classification
- direct parent relationships
- direct child relationships when useful
- direct broader or narrower relationships
- direct typed edges such as `grounds`, `aligns_with`, `constrains`, or `derived_from`
- provenance and review state
- trust and boundary classifications where relevant

## Machine-derived layer

Machines may infer from governed local assertions:
- ancestor chains
- descendant chains
- transitive broader or narrower traversals
- local-to-global graph neighborhoods
- concordance expansions
- grouped retrieval clusters
- repeated branch patterns and candidate promotion signals

## What machines should not infer casually

Machines should not silently invent:
- new doctrinal conclusions
- new trust or heresy classifications
- new canonical status claims
- new controlled relationship verbs
- new node types or schema grammars
- high-confidence semantic links without governed support

Inference should expand from governed structure, not replace it.

## Direct vs transitive rule

When possible:
- humans should record direct local structure
- machines should compute transitive closure or expanded maps

This helps keep authored nodes lean and stable while allowing the repository to scale in machine traversal.

## Trust rule

Inference should respect trust and boundary layers.

That means machines should not flatten together:
- canonical and noncanonical sources
- trusted and disputed translations
- reviewed and community-proposed objects
- strong doctrinal grounding and weak associative linkage

Inference must preserve the repository's boundary logic.

## Review rule

When a machine-generated pattern appears important enough to reuse, a human should decide whether to:
- keep it as a derived result only
- promote it into a governed node
- promote it into a relationship object
- reject it as noise or overreach

## Fractal rule

Inference policy should itself remain fractal.

That means the same general rule should recur across levels:
- local nodes stay lean
- branch structures stay governed
- graph layers become richer through inference
- promotion happens when repeated derived structure becomes valuable enough to stabilize

## Summary principle

Humans author local truth.
Machines infer larger traversable structure.

That division is one of the strongest ways to keep the repository both fractal and governable.
