# Feedback and Learning Loops

## Purpose

This file defines how the Logos repository should improve over time through structured feedback rather than ungoverned drift.

## Core principle

The repository should become self-improving in workflow, not self-authorizing in doctrine.

That means feedback should improve:
- proposal quality
- validation quality
- review routing
- retrieval usefulness
- structural coherence

without allowing the system to silently redefine theology.

## Feedback categories

### 1. Structural feedback
Track:
- broken references
- bad addresses
- duplicate concepts
- orphan nodes
- invalid claims
- unresolved migrations

### 2. Review feedback
Track:
- approval rates
- revision rates
- rejection rates
- quarantine rates
- average time to review
- branches generating the most rejected AI proposals

### 3. Retrieval feedback
Track:
- repeated failed queries
- repeated missing-context patterns
- nodes often retrieved together
- missing explanatory nodes that cause retrieval weakness

### 4. Governance feedback
Track:
- trust promotions later reversed
- boundary-classification confusion
- unstable relationship verbs
- repeated overlay-scope confusion

## Learning rule

The system may use feedback to improve:
- proposal templates
- validation rules
- review queue routing
- recommendation quality

But feedback must not silently:
- promote speculative material into core status
- erase doctrinal boundaries
- flatten trust zones
- override human review decisions

## Summary principle

The Logos repository should learn structurally and operationally from feedback while preserving explicit human authority over doctrinal and boundary-sensitive decisions.
