# Build Order Policy

## Purpose

This file defines the required order for building the Logos repository so that governance, structure, and validation mature before large-scale expansion.

## Core rule

Do not scale the content surface faster than the governing grammar.

A strong build order reduces drift, duplication, and unsafe AI-assisted expansion.

## Required build order

### Phase 1. Governance stabilization
Write or reconcile:
- continuation guidance
- self-building rules
- human-in-the-loop review rules
- trust and boundary expectations

### Phase 2. Structural address system
Define:
- structural address format
- relationship between ID and address
- migration rules when address changes

### Phase 3. Claim layer
Define:
- claim object grammar
- asserted / inferred / hypothesis separation
- evidence and lineage expectations

### Phase 4. Root shell normalization
Normalize root branches so serious branches share a consistent metadata grammar.

### Phase 5. Validation and CI
Define machine-checkable structure before broad buildout.

### Phase 6. Feedback and drift detection
Define how the system will detect structural, review, and retrieval failure.

### Phase 7. Reference implementation slice
Build one full vertical slice before broad expansion.

### Phase 8. Overlay logic
Add shared-core, tradition, institution, and application overlay handling.

### Phase 9. Broader content expansion
Scale branch content only after the earlier layers are sufficiently stable.

### Phase 10. Aggressive automation
Increase automation only after governance, validation, and review pathways are operational.

## Anti-patterns

Avoid:
- building many branches in parallel before the metadata shell is stable
- introducing address changes without a migration model
- expanding graph surfaces without a claim layer
- letting AI generate large amounts of doctrine-shaped content without explicit review routing
- building overlays before the shared core is clear enough to support them

## Practical sequence

The correct working rule is:
- reconcile first
- validate second
- expand third
- automate fourth

## Summary principle

Build the Logos repository in layers.
Do not let faster content growth outrun governance, identity, review, and validation discipline.
