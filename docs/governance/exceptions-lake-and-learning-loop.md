# Exceptions Lake and Learning Loop

## Purpose

This file defines the native exceptions-lake layer for the Logos repository.

The exceptions lake is where reality resists the current model in governed, append-only form.
It preserves pressure, mismatch, and unresolved evidence without silently mutating doctrine, taxonomy, or governance.

## Distinctions that must remain explicit

- **Ontology** = what the system currently believes.
- **Taxonomy** = how the system organizes use.
- **Exceptions lake** = where observed reality does not fit current expectations.
- **Learning loop** = how reviewed exceptions may be promoted into governed change.

The architecture should not collapse these into one artifact.

## Canonical nine-layer exception address

The exceptions-lake layer uses this canonical address grammar:

`root.domain.capability.entity.instance.state.expectation.exception.adaptation`

### Layer intent

1. `root` — stable top-level project root.
2. `domain` — governed branch domain (theology, scripture, graph, application, etc.).
3. `capability` — decision or operational capability under review.
4. `entity` — actor, process, model, institution, or theological object being evaluated.
5. `instance` — concrete run, case, event, or slice.
6. `state` — observed local state at capture time.
7. `expectation` — declared threshold, rule, or normative target.
8. `exception` — mismatch between expectation and observed state.
9. `adaptation` — proposed or approved response path.

### Non-collapse rule

`expectation`, `exception`, and `adaptation` are separate governed objects even when linked.

- Expectation records what should have happened.
- Exception records what actually resisted the model.
- Adaptation records what to change (or decline to change) after review.

## AIRCA bridge for exception handling

The exceptions-lake loop aligns to AIRCA/LAIRCA stages:

- **Architect** -> define expectations, thresholds, and failure boundaries.
- **Inform** -> capture observations, evidence, and exception events.
- **Rank** -> score exception severity, recurrence, blast radius, and doctrinal risk.
- **Commit** -> approve, reject, defer, or quarantine promotion actions.
- **Act** -> apply governed changes to rules, taxonomy, ontology, or governance docs.

This keeps exception learning operationally compatible with AIRCA while preserving Logos theological upstream discipline.

## Governance layering for promotion review

Every promotable adaptation should be assessed through the existing governance layers:

1. worldview
2. institutional
3. operating
4. human
5. technical

A proposal that passes technical criteria but fails worldview or human-accountability criteria should not be auto-promoted.

## Pressure-vector lenses

Each exception should be classifiable under one or more pressure vectors:

- theological
- psychological
- moral
- political
- economic
- cultural

Pressure vectors are analysis lenses, not replacement taxonomies.
They help route review, compare recurrence patterns, and detect where taxonomy or ontology is under strain.

## Promotion path: exception to governed change

Use the following append-only path:

1. **Capture** exception event with evidence and nine-layer address.
2. **Classify** pressure vectors and affected governance layers.
3. **Score** priority and risk (Rank stage).
4. **Review** with human accountability (Commit stage).
5. **Promote** one approved action type:
   - rule change
   - taxonomy update
   - ontology update
   - governance protocol update
6. **Link lineage** from adaptation to changed object(s).
7. **Retain source record** of the original exception event.

Promotion is additive and traceable. Historical exception records remain intact.

## Provenance, trust, and graph alignment

Exceptions-lake objects should preserve:

- source evidence references
- capture actor/process identity
- review status
- trust-zone classification
- supersedes/superseded-by lineage

When modeled machine-readably, exceptions should live in governed graph schemes and templates rather than ad hoc files.
