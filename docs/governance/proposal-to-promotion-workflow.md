# Proposal-to-Promotion Workflow

## Purpose

This file defines how candidate repository objects move from proposal into governed reuse.

## Core principle

Important objects should not appear as silently accepted truth.
They should pass through a governed workflow.

## Workflow

### 1. Detect
A gap, repeated pattern, or retrieval weakness is detected.

### 2. Propose
A candidate node, claim, edge object, or retrieval package is drafted.

### 3. Validate
Automated validation checks structure, required fields, reference integrity, and approved field values.

### 4. Queue
The candidate enters a review queue appropriate to its branch or sensitivity.

### 5. Review
A human reviewer marks the candidate as:
- approved
- approved_with_revision
- rejected
- quarantined
- deferred

### 6. Promote
Approved objects are promoted into the governed live architecture with explicit lifecycle and trust posture.

### 7. Measure
Feedback loops assess whether the promoted object actually improved retrieval, coherence, or coverage.

## Review rule

High-sensitivity proposals require human review before promotion.
Examples include doctrinal classification, boundary classification, trust-zone promotion, and major synthesis claims.

## Summary principle

A proposal becomes reusable repository structure only after validation and review.
